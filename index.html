The Problem

Three times a year, benchmark testing produced hundreds of student results. All of them had to be filtered, formatted, and organized into parent communication packets before anything could go home to families.

Teachers prepared students for the testing. Literacy specialists handled everything that came after: administering the assessments, generating the parent letters, tracking what went out. Working closely with both groups, I found myself in a position to see the entire workflow instead of just one piece of it.

The fix didn’t arrive all at once. It grew in stages, each one replacing a little more of the manual work:

Manual process → mail merge → cleaner workflow → automation → AI assistance → standalone application

There was also a promise in the middle of it. I’d told a friend and colleague I would always take care of the letters, and I meant it.

The breakthrough wasn’t building software. It was realizing the only way to keep that promise was to make the process stop depending on me.

That’s what turned a collection of automations into something anyone can run. Not another spreadsheet. A spreadsheet designed to behave like an application.

The Product

A teacher exports their benchmark data, drops the file in a folder, and clicks one button. The system imports the newest file, filters to the selected assessment window, groups students by teacher, and generates a formatted Google Doc packet for each one.

How It Works

The system is a pipeline, not a single script. Raw data is imported, cleaned, and validated into one processed dataset that acts as the single source of truth. The workflow engine reads only from that dataset: it selects students, merges data into templates, and generates a packet for each teacher.

Built to extend

The split ending is deliberate. Parent letters are live today. Student reports, MTSS summaries, and compliance documents are planned modules that hang off the same engine. New requirements become an output-layer change, not a new project.

Engineering Decisions
The constraint

Google Apps Script stops any single execution at about six minutes. A full building’s worth of letters takes longer than that, so a naive version would quit partway through.

Workspace accounts get a longer window, but I built for the shorter limit on purpose so the tool runs on any account. Then I designed around the constraint instead of against it.

The workflow processes as many packets as fit in a safe time budget, saves its progress in the data itself, and schedules its own continuation. It resumes where it left off until every packet is complete. Because progress lives in the data, it can also recover if anything interrupts it mid-run.

This turned “keep clicking until it finishes” into “click once and walk away.”

Reporting, not deciding

The tool reports each student’s benchmark results. It doesn’t interpret them or recommend what to do next. What the data means for a child stays with educators, and future modules will hold that same line.

The time saved is the obvious win. The bigger one is reliability. The mistakes that used to be easy to make, like the wrong window or a broken merge field, can’t happen anymore.

A solution that only one person can run isn’t really a solution.

One that other people can pick up and run on their own is. Early on, I documented a simpler mail-merge version of the process, and other schools began running their own copies. The approach spread well beyond my building, and I have a feeling this version will make it to them too.

Leverage where it counts

The district has since merged the reading and math specialist roles into one position. One person now carries the workload two people used to share. A tool that turns days of work into one click is what makes that survivable.

What’s Next
More output modules

Administrator reports, intervention documentation, and compliance notifications, all on the same engine.

A hardened relay

A continuation mechanism fully resilient to slow network conditions.

Built to adapt

The district is moving away from one assessment subject. That’s an output-layer change, not a rebuild.

The goal was a polished, documented version that solves a real problem well, not the largest possible application.

Where AI Fit

AI accelerated the build. It didn’t replace the thinking. The problem definition, the architecture, the tradeoffs, and the testing were mine.
